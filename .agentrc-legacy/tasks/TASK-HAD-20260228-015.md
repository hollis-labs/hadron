## Description

Port the MCP adapter from vnext, renaming all tools from `cortex_*` to `hadron_*` and adding the missing `hadron_blueprint_validate` tool.

## Instructions

- Copy `vnext-blueprint-runner/internal/mcpadapter/adapter.go` → `hadron/internal/mcpadapter/adapter.go`
- Copy `vnext-blueprint-runner/internal/mcpadapter/tools.go` → `hadron/internal/mcpadapter/tools.go`
- Copy `vnext-blueprint-runner/internal/mcpadapter/tools_test.go` → `hadron/internal/mcpadapter/tools_test.go`
- Update import paths: `hollis-labs/cortex` → `hollis-labs/hadron`
- Rename ALL tool names: `cortex_*` → `hadron_*` (global replace in tools.go)
- Rename all tool descriptions: remove "Cortex" references, use "Hadron"
- Add new tool: `hadron_blueprint_validate`
  - Input: `content` (string, required) — YAML/JSON blueprint content
  - Input: `format` (string, optional) — "yaml" or "json", auto-detected if absent
  - Calls `blueprint.ParseBytes([]byte(content))` internally
  - Returns: `{"valid": true}` or `{"valid": false, "error": "..."}`
- Verify `go test ./internal/mcpadapter/...` passes

**Full tool list after rename** (see architecture doc for descriptions):
`hadron_health`, `hadron_workspaces_list`, `hadron_workspace_get`, `hadron_workspace_create`,
`hadron_runs_list`, `hadron_run_get`, `hadron_run_enqueue`, `hadron_run_cancel`, `hadron_run_events`,
`hadron_schedules_list`, `hadron_schedule_create`, `hadron_schedule_update`,
`hadron_pipelines_list`, `hadron_pipeline_enqueue`, `hadron_pipeline_stages`,
`hadron_blueprint_validate` (new)

## Acceptance

- All tool names start with `hadron_`
- `go test ./internal/mcpadapter/...` passes
- `hadron_blueprint_validate` correctly validates a YAML blueprint
- `hadron_blueprint_validate` returns structured error for invalid blueprint
- MCP adapter starts in stdio mode without panicking
