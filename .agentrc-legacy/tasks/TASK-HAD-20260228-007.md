## Description

Implement the execution manager — the heart of Hadron. Executes blueprint tasks via PTY, handles retry/timeout/conditions/hooks, persists events to SQLite, and emits structured events to callers.

## Instructions

Build `hadron/internal/execution/manager.go` combining:
- Worker pool architecture from vnext `execution/manager.go` (Request/Job model, Enqueue interface)
- PTY execution from wails-starter `core/manager.go` (pty.Start, bufio.Scanner, retry loops, timeout with context)
- SQLite event persistence (persist events to `run_events` table via `persistence.Store`)

**`Request` struct:**
```go
type Request struct {
    WorkspaceID   string
    RunID         string
    BlueprintPath string
    Inputs        map[string]any
    DryRun        bool
}
```

**`Manager`** should:
- Accept a worker count, persistence.Store, and settings.Settings
- Expose `Enqueue(ctx, Request) error`
- On enqueue: create RunRecord in persistence, add to job queue
- Workers: pick up jobs, set run started, execute all sections/tasks sequentially
- On job complete: set run finished (done or failed)

**Step execution:**
- Render blueprint for execution (template context with inputs, env, project, workspace)
- For each section, for each task:
  - Check `enabled` (skip if false)
  - Evaluate `if`/`condition` (render template, then if not "true"/"false" run as shell command)
  - Validate command via settings
  - Validate dir via settings
  - Execute via PTY (`bash -lc <cmd>`)
  - Stream lines to `run_events` table (event_type: "log")
  - Handle retry loop with delay
  - Handle timeout via context.WithTimeout
  - On success: execute `on_success` hooks
  - On fail: execute `on_fail` hooks; if not continue_on_error, halt job
- Hook types: cmd (exec), error (emit error event), step (jump to named task), blueprint (load+exec external), call (exec imported blueprint by alias)

**DryRun mode:** validate and render blueprint, log what would be executed, do not exec any commands

**Telemetry:** port telemetry.Logger from wails-starter, wire into Manager for structured job start/end/task events

## Acceptance

- `go build ./internal/execution/...` passes
- Manager processes a job with 3 sections, 2 tasks each — all events persisted to DB
- Retry: task fails, retries N times, events show retry log lines
- Timeout: task exceeds timeout_seconds, job marked failed with timeout error
- Condition false: task skipped, skip event persisted
- DryRun: no commands executed, validation passes
- `on_success` cmd hook runs after successful task
- `on_fail` error hook marks job failed when continue_on_error=false
